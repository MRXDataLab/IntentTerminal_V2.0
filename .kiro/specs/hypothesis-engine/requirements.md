# Requirements Document

## Introduction

The Hypothesis Engine elevates hypothesis generation in the Outtlyr Nucleus Ingestion Platform (Module 1) from a scattered byproduct of multiple stages into a dedicated first-class stage. The engine consumes the full intake pillar payload after intent lock and emits a structured Hypothesis Manifest that becomes the single source of truth for the Strategic Brief, Link Farming Manifest, Category Graph, Methodology Blueprint, and Intelligence Map.

The engine introduces three internal stages — Decomposition, Generation, and MECE Audit — produces explicit contrarian pairs to counter confirmation bias, runs generation twice per problem (with and without chat-history context) to surface unknown unknowns, and exposes the manifest through a new API endpoint and a new frontend review phase inserted between Intake and Synthesis.

The Intake Terminal's Pillar 3 is refactored to capture decision goals and success criteria rather than asking the client for a hypothesis directly.

## Glossary

- **Outtlyr_Platform**: The Outtlyr Nucleus Ingestion Platform (Module 1) that this update modifies.
- **Hypothesis_Engine**: The new backend service implemented at `backend/services/hypothesis_engine.py` that generates the Hypothesis Manifest.
- **Hypothesis_Manifest**: The machine-readable JSON artifact produced by the Hypothesis_Engine, persisted as `Hypothesis_Manifest.json`, conforming to schema version 1.0.
- **Decomposition_Stage**: The first internal stage of the Hypothesis_Engine that parses intent and pillars into one or more Core Problems via a single LLM call.
- **Generation_Stage**: The second internal stage of the Hypothesis_Engine that generates hypotheses and contrarian pairs per Core Problem via two LLM calls per problem (context-aware and naive).
- **MECE_Auditor**: The third internal stage of the Hypothesis_Engine that audits the hypothesis set for semantic overlap via a single LLM call with fresh context.
- **Core_Problem**: A discrete problem decomposed from the research intent; each Core Problem has its own hypothesis sub-tree under the shared manifest.
- **Hypothesis**: A single structured hypothesis record within the Hypothesis_Manifest with fields including `id`, `statement`, `dimension`, `force_assignment`, `expected_signals`, and `contrarian_pair_id`.
- **Contrarian_Pair**: Two hypotheses within the same dimension that explicitly state opposite positions and reference each other through `contrarian_pair_id`.
- **MECE**: Mutually Exclusive, Collectively Exhaustive — the property the MECE_Auditor enforces across the hypothesis set.
- **Intake_Terminal**: The frontend component `frontend/src/components/IntakeTerminal.tsx` that handles the five-pillar intake conversation.
- **Hypothesis_Review_Screen**: The new frontend component `frontend/src/components/HypothesisReview.tsx` rendered during the `hypothesis_generation` phase.
- **Synthesis_Dashboard**: The existing frontend component `frontend/src/components/SynthesisDashboard.tsx` that consumes the manifest after Hypothesis_Review_Screen approval.
- **Strategic_Brief**: The artifact generated by `backend/api/brief.py`.
- **Link_Farming_Manifest**: The artifact generated by `backend/api/manifest.py`.
- **Category_Graph**: The graph artifact generated by `backend/api/ecosystem.py`.
- **Intelligence_Map**: The artifact generated by `backend/api/intelligence_map.py`.
- **Phase_State_Machine**: The frontend phase state in `frontend/src/app/page.tsx` with values `interaction`, `hypothesis_generation`, `synthesis`, and `discovery`.
- **Dev_Bypass**: The development-mode shortcut buttons that skip phases by loading cached `latest_run_data` artifacts.
- **Pillar_3**: The third pillar of the Intake_Terminal, renamed from "Target Lens & Hypothesis" to "Target Lens & Decision Goal".
- **Hypothesis_Anchor**: A new node type in the Category_Graph rendered with color `#6366f1` (Indigo) and size 6, representing a hypothesis from the manifest.
- **Strategic_Force**: One of the five strategic forces defined in `backend/services/force_taxonomy.py`, assigned to each hypothesis via `force_assignment`.
- **Structural_Dimension**: One of ten taxonomy dimensions used by the Generation_Stage: `price`, `product`, `brand_perception`, `distribution`, `cultural_identity`, `regulatory`, `demographic_shift`, `competitive_shift`, `situational_context`, `identity_expression`.
- **De_Anchored_Generation**: The practice of running the Generation_Stage twice per Core Problem — once with full pillar context and once with only the intent string and no chat history — and merging the two outputs.
- **Hard_Validation_Rules**: The five validation rules defined for the Hypothesis_Manifest in Section 5.2 of the source specification.

## Requirements

### Requirement 1: Hypothesis Engine Service

**User Story:** As a research analyst, I want a dedicated backend service that generates a Hypothesis_Manifest from the intake payload, so that all downstream stages consume the same structured hypothesis source of truth.

#### Acceptance Criteria

1. THE Hypothesis_Engine SHALL expose a public function `generate_hypothesis_manifest` that accepts `intent`, `pillar_extractions`, `pillar_scores`, `context_document`, `template`, and `chat_history` as inputs and returns a Hypothesis_Manifest dict conforming to schema version 1.0.
2. THE Hypothesis_Engine SHALL execute three internal stages in sequence: Decomposition_Stage, Generation_Stage, and MECE_Auditor.
3. THE Hypothesis_Engine SHALL execute statelessly with no module-level mutable globals.
4. THE Hypothesis_Engine SHALL persist the produced Hypothesis_Manifest to `backend/Hypothesis_Manifest.json` and to `backend/latest_run_data/latest_hypothesis_manifest.json` after successful generation.

### Requirement 2: Decomposition Stage

**User Story:** As a research analyst, I want the engine to decompose multi-problem intents into discrete core problems, so that each distinct problem receives its own hypothesis sub-tree.

#### Acceptance Criteria

1. WHEN `generate_hypothesis_manifest` is invoked, THE Decomposition_Stage SHALL parse the intent and pillar extractions into an array of one or more Core_Problem records via a single LLM call.
2. THE Decomposition_Stage SHALL assign each Core_Problem a unique `id` and a `priority` value of either `primary` or `secondary`.
3. THE Decomposition_Stage SHALL set `decomposed_from_intent` to true on every Core_Problem record returned.
4. IF the Decomposition_Stage returns zero Core_Problem records, THEN THE Hypothesis_Engine SHALL treat the manifest as invalid and trigger regeneration per Requirement 6.

### Requirement 3: Generation Stage with De-Anchored Pass

**User Story:** As a research analyst, I want the engine to generate hypotheses twice per problem — once with full context and once without chat history — so that confirmation bias from the intake conversation does not suppress unknown unknowns.

#### Acceptance Criteria

1. FOR each Core_Problem returned by the Decomposition_Stage, THE Generation_Stage SHALL execute two LLM calls: one context-aware call that receives the full pillar payload and chat history, and one naive call that receives only the intent string with no chat history.
2. THE Generation_Stage SHALL merge the context-aware and naive outputs into a single hypothesis list per Core_Problem.
3. THE Generation_Stage SHALL set the `generation_source` field of each hypothesis to one of `context_aware`, `naive`, or `contrarian_pair`.
4. THE Generation_Stage SHALL evaluate the problem space against all ten Structural_Dimensions and SHALL list every dimension considered in `metadata.dimensions_covered`.
5. FOR every hypothesis generated, THE Generation_Stage SHALL generate an explicit contrarian pair within the same `dimension` and SHALL cross-reference both records through `contrarian_pair_id`.
6. WHERE a hypothesis has no meaningful semantic opposite, THE Generation_Stage SHALL set its `contrarian_pair_id` to null and SHALL include a justification for the absent pair in the `rationale` field.
7. THE Generation_Stage SHALL assign each hypothesis a `force_assignment` value drawn from the five Strategic_Force values defined in `backend/services/force_taxonomy.py`.
8. THE Generation_Stage SHALL assign each hypothesis a minimum of three values in `expected_signals` drawn from the canonical signal taxonomy.
9. THE Generation_Stage SHALL assign each hypothesis an `investigation_priority` value of `high`, `medium`, or `low`.

### Requirement 4: MECE Audit Stage

**User Story:** As a research analyst, I want the engine to audit and dedupe overlapping hypotheses, so that the manifest contains mutually exclusive entries.

#### Acceptance Criteria

1. WHEN the Generation_Stage completes for all Core_Problem records, THE MECE_Auditor SHALL execute a single LLM call with fresh context and no chat history that reviews the full hypothesis list.
2. THE MECE_Auditor SHALL compute pairwise semantic similarity for hypotheses sharing the same `dimension`.
3. WHEN two hypotheses share the same `dimension` and have semantic similarity greater than 75 percent and are not declared contrarian pairs of each other, THE MECE_Auditor SHALL flag the pair and SHALL emit a merge directive specifying the merged statement.
4. THE Hypothesis_Engine SHALL apply MECE_Auditor merge directives by combining flagged hypotheses into a single broader hypothesis record.
5. THE Hypothesis_Engine SHALL record the number of audit passes executed in `metadata.mece_audit_iterations`.
6. WHEN the MECE_Auditor completes without flagging any non-contrarian overlap, THE Hypothesis_Engine SHALL set `metadata.mece_audit_passed` to true.
7. IF the MECE_Auditor still flags overlap after three iterations, THEN THE Hypothesis_Engine SHALL set `metadata.mece_audit_passed` to false and SHALL still return the manifest.

### Requirement 5: Hypothesis Manifest Schema

**User Story:** As a downstream consumer of the manifest, I want a stable machine-readable schema, so that the Strategic_Brief, Link_Farming_Manifest, Category_Graph, and Intelligence_Map can consume the manifest programmatically.

#### Acceptance Criteria

1. THE Hypothesis_Manifest SHALL include `schema_version` set to the string `"1.0"`.
2. THE Hypothesis_Manifest SHALL include `generated_at` as an ISO 8601 timestamp string.
3. THE Hypothesis_Manifest SHALL include `intent` as the original intent string supplied to the engine.
4. THE Hypothesis_Manifest SHALL include a `metadata` object containing `total_core_problems`, `total_hypotheses`, `total_contrarian_pairs`, `generation_method`, `mece_audit_passed`, `mece_audit_iterations`, `de_anchored_pass_count`, and `dimensions_covered`.
5. THE Hypothesis_Manifest SHALL include a `core_problems` array where each element contains `id`, `statement`, `priority`, `decomposed_from_intent`, `hypothesis_count`, and `hypotheses`.
6. THE Hypothesis_Manifest SHALL ensure each hypothesis record contains `id`, `statement`, `dimension`, `force_assignment`, `mece_cluster_id`, `expected_signals`, `expected_counter_signals`, `contrarian_pair_id`, `investigation_priority`, `generation_source`, and `rationale`.
7. THE Hypothesis_Manifest SHALL constrain `hypotheses[].dimension` to one of the ten Structural_Dimension values listed in the Glossary.

### Requirement 6: Hard Validation and Regeneration

**User Story:** As a platform operator, I want invalid manifests to trigger automatic regeneration with stricter constraints, so that downstream stages never receive malformed inputs.

#### Acceptance Criteria

1. THE Hypothesis_Engine SHALL classify a manifest as invalid IF any hypothesis has fewer than three values in `expected_signals`.
2. THE Hypothesis_Engine SHALL classify a manifest as invalid IF any hypothesis lacks a `contrarian_pair_id` and the `rationale` does not include a justification for the absent pair.
3. THE Hypothesis_Engine SHALL classify a manifest as invalid IF two hypotheses share the same `mece_cluster_id` and neither hypothesis is the contrarian pair of the other.
4. THE Hypothesis_Engine SHALL classify a manifest as invalid IF `total_core_problems` is zero.
5. THE Hypothesis_Engine SHALL classify a manifest as invalid IF any Core_Problem contains zero hypotheses.
6. WHEN a generated manifest is classified as invalid, THE Hypothesis_Engine SHALL retry generation with stricter constraints up to three times before returning.
7. IF all three regeneration attempts fail to produce a valid manifest, THEN THE Hypothesis_Engine SHALL surface the failure to the API layer per Requirement 7.

### Requirement 7: Hypothesis Generation API Endpoint

**User Story:** As the frontend, I want a dedicated POST endpoint that returns a Hypothesis_Manifest, so that the Hypothesis_Review_Screen can render the engine's output.

#### Acceptance Criteria

1. THE Outtlyr_Platform SHALL expose `POST /api/generate-hypotheses` accepting a JSON body containing `intent`, `pillar_extractions`, `pillar_scores`, `context_document`, `template`, and `chat_history`.
2. WHEN the endpoint receives a valid request, THE Outtlyr_Platform SHALL invoke the Hypothesis_Engine and SHALL return the resulting Hypothesis_Manifest as the response body with HTTP status 200.
3. IF the request payload is missing required pillars or contains an invalid intent, THEN THE Outtlyr_Platform SHALL return HTTP status 400 with an error description.
4. IF the Hypothesis_Engine fails all three regeneration attempts to produce a valid manifest, THEN THE Outtlyr_Platform SHALL return HTTP status 422 with an error description.
5. IF the LLM provider rate-limits the request and the cascading model fallback is exhausted, THEN THE Outtlyr_Platform SHALL return HTTP status 429.
6. IF the Hypothesis_Engine raises an internal error, THEN THE Outtlyr_Platform SHALL return HTTP status 500 and SHALL log the error.

### Requirement 8: Knowledge Base Agent for the Engine

**User Story:** As a developer maintaining engine prompts, I want a single hot-reloadable KB file that contains all three engine sub-prompts, so that prompt changes do not require redeployment.

#### Acceptance Criteria

1. THE Outtlyr_Platform SHALL contain a knowledge base file at `backend/kb/agents/hypothesis_agent.md` containing three sub-prompt sections labeled `[DECOMPOSER]`, `[GENERATOR]`, and `[MECE_AUDITOR]`.
2. THE Hypothesis_Engine SHALL load the appropriate sub-prompt by stage selection from the knowledge base file.
3. THE Outtlyr_Platform SHALL load the knowledge base file through the existing `kb_loader.py` mtime-cache pattern so that changes to `hypothesis_agent.md` are picked up without restart.

### Requirement 9: Phase State Machine Update

**User Story:** As a frontend user, I want a new `hypothesis_generation` phase between intake and synthesis, so that the manifest is reviewed before downstream artifacts are generated.

#### Acceptance Criteria

1. THE Phase_State_Machine SHALL accept the values `interaction`, `hypothesis_generation`, `synthesis`, and `discovery`.
2. WHEN the Intake_Terminal completes with `is_finalized: true`, THE Outtlyr_Platform SHALL transition the Phase_State_Machine from `interaction` to `hypothesis_generation`.
3. THE Outtlyr_Platform SHALL preserve `interactionPayload` state across the transition into and out of `hypothesis_generation` without wiping the payload.
4. THE Outtlyr_Platform SHALL maintain a frontend state field `hypothesisManifest` that persists across rejection loops from later phases using the same persistence pattern as `interactionPayload`.

### Requirement 10: Hypothesis Review Screen

**User Story:** As a research analyst, I want a review screen that surfaces the generated hypotheses with counts, contrarian pairs, and audit status, so that I can inspect the engine's output before synthesis runs.

#### Acceptance Criteria

1. WHILE `phase` equals `hypothesis_generation` and the Hypothesis_Manifest has been successfully generated, THE Outtlyr_Platform SHALL render the Hypothesis_Review_Screen.
2. THE Hypothesis_Review_Screen SHALL display a counter row containing the total core problem count, total hypothesis count, total MECE cluster count, and MECE audit status.
3. THE Hypothesis_Review_Screen SHALL render one section per Core_Problem with the problem statement and priority label.
4. THE Hypothesis_Review_Screen SHALL render each hypothesis as a card displaying the hypothesis ID, dimension chip, investigation priority chip, full statement, force assignment, expected signal count, contrarian pair reference, and a collapsible rationale.
5. WHEN the user clicks a contrarian pair reference on a hypothesis card, THE Hypothesis_Review_Screen SHALL scroll the viewport to the paired hypothesis card.
6. THE Hypothesis_Review_Screen SHALL display an action bar containing the buttons "Proceed to Synthesis", "Add Hypothesis", "Edit", and "Regen".
7. THE Hypothesis_Review_Screen SHALL highlight the "Proceed to Synthesis" button as the default action.
8. WHEN the user clicks "Proceed to Synthesis", THE Outtlyr_Platform SHALL persist the manifest and SHALL transition the Phase_State_Machine to `synthesis`.
9. WHEN the user clicks "Add Hypothesis", THE Hypothesis_Review_Screen SHALL open a modal that requires the user to specify dimension, force, expected signals, and contrarian pair before submission.
10. WHEN the user clicks "Regen", THE Outtlyr_Platform SHALL invoke `POST /api/generate-hypotheses` with optional refinement context.
11. WHEN the user rejects the manifest from the Hypothesis_Review_Screen, THE Outtlyr_Platform SHALL transition the Phase_State_Machine back to `interaction` with rejection context attached.

### Requirement 11: MECE Audit Failure Banner

**User Story:** As a research analyst, I want a clear visual warning when the MECE audit did not converge, so that I know to inspect overlapping hypotheses before proceeding.

#### Acceptance Criteria

1. WHEN `metadata.mece_audit_passed` is false, THE Hypothesis_Review_Screen SHALL display a yellow warning banner at the top of the screen describing that the audit did not converge after three iterations.
2. WHEN `metadata.mece_audit_passed` is false, THE Hypothesis_Review_Screen SHALL render a dashed connector linking each pair of hypotheses flagged as overlapping.
3. WHEN the user hovers over a flagged hypothesis pair connector, THE Hypothesis_Review_Screen SHALL display a popover containing the MECE_Auditor's merge suggestion text.

### Requirement 12: Intake Pillar 3 Refactor

**User Story:** As a client using the Intake_Terminal, I want Pillar_3 to ask about my decision goal and success criteria rather than my hypothesis, so that the engine can generate the hypothesis space without confirmation bias from my framing.

#### Acceptance Criteria

1. THE Intake_Terminal SHALL display the Pillar_3 label as `"Target Lens & Decision Goal"`.
2. THE Intake_Terminal SHALL score Pillar_3 saturation based on capture of decision context, success criteria, and risk tolerance.
3. THE Intake_Terminal SHALL NOT ask the client to articulate a hypothesis during Pillar_3 conversation.
4. THE Intake_Terminal SHALL set `is_finalized` to true without requiring hypothesis articulation in any pillar.
5. THE Outtlyr_Platform SHALL update `backend/kb/agents/intake_agent.md` to remove hypothesis-elicitation language and to instruct the LLM that hypothesis generation is performed by a downstream engine.
6. THE Outtlyr_Platform SHALL update each of the six archetype files in `backend/kb/agents/intake_archetypes/` (`ua.md`, `brand_health.md`, `market_entry.md`, `competitive_pulse.md`, `erosion_study.md`, `pricing_value.md`) to remove hypothesis-elicitation language.
7. THE Intake_Terminal SHALL retain the existing scoring logic, labels, and saturation criteria for the four pillars other than Pillar_3.
8. THE Intake_Terminal SHALL retain the existing readiness label thresholds: 0–20 VAGUE, 21–40 EMERGING, 41–60 DEVELOPING, 61–70 CALIBRATING, 71–100 SATURATED.

### Requirement 13: Strategic Brief Manifest Consumption

**User Story:** As a downstream consumer, I want the Strategic_Brief to render hypotheses from the manifest rather than generate new ones, so that the brief and the manifest never drift apart.

#### Acceptance Criteria

1. THE Strategic_Brief endpoint `POST /api/generate-brief` SHALL accept an optional `hypothesis_manifest` parameter in the request body.
2. WHEN `hypothesis_manifest` is provided to `POST /api/generate-brief`, THE Strategic_Brief SHALL render its "Hypotheses" section as a structured render of the supplied manifest with one subsection per Core_Problem.
3. THE Strategic_Brief SHALL render each hypothesis with the statement, dimension, force assignment, expected signals, and a rationale paragraph.
4. THE Strategic_Brief SHALL render contrarian pairs side-by-side within the same Core_Problem subsection.
5. WHEN `hypothesis_manifest` is provided, THE Strategic_Brief SHALL NOT generate new hypotheses inline.
6. WHERE `hypothesis_manifest` is absent from the request, THE Strategic_Brief SHALL retain its current behavior of generating hypotheses inline.
7. THE Outtlyr_Platform SHALL update `backend/kb/agents/brief_agent.md` to instruct the LLM to render the supplied manifest as prose rather than generate new hypotheses.

### Requirement 14: Link Farming Manifest Consumption

**User Story:** As a downstream consumer, I want the Link_Farming_Manifest to derive search nets from the manifest's expected signals, so that crawling targets reflect the hypothesis space.

#### Acceptance Criteria

1. THE Link_Farming_Manifest endpoint `POST /api/generate-manifest` SHALL accept an optional `hypothesis_manifest` parameter in the request body.
2. WHEN `hypothesis_manifest` is provided to `POST /api/generate-manifest`, THE Link_Farming_Manifest SHALL translate each hypothesis's `expected_signals` into search nets.
3. WHEN `hypothesis_manifest` is provided to `POST /api/generate-manifest`, THE Link_Farming_Manifest SHALL use each hypothesis's `dimension` to inform platform routing.
4. WHERE `hypothesis_manifest` is absent from the request, THE Link_Farming_Manifest SHALL retain its current behavior.

### Requirement 15: Category Graph Consumption

**User Story:** As a downstream consumer, I want the Category_Graph to render hypotheses as first-class anchor nodes, so that the graph reflects the hypothesis space as primary structure.

#### Acceptance Criteria

1. THE Category_Graph endpoint `POST /api/generate-ecosystem` SHALL accept an optional `hypothesis_manifest` parameter in the request body.
2. WHEN `hypothesis_manifest` is provided to `POST /api/generate-ecosystem`, THE Category_Graph SHALL include a node of type `hypothesis_anchor` for each hypothesis in the manifest.
3. THE Category_Graph SHALL render each `hypothesis_anchor` node with color `#6366f1` and size 6.
4. THE Category_Graph SHALL attach each `hypothesis_anchor` node to the Core_Problem node corresponding to the hypothesis's parent Core_Problem.
5. THE Category_Graph SHALL attach each Core_Problem node to the root node.
6. THE Category_Graph SHALL hang the existing Subject, Component, and Signal hierarchy off each `hypothesis_anchor` node.
7. WHERE `hypothesis_manifest` is absent from the request, THE Category_Graph SHALL retain its current behavior.

### Requirement 16: Intelligence Map Consumption

**User Story:** As a downstream consumer, I want the Intelligence_Map to populate `explicit_hypothesis` nodes from the manifest rather than regenerate them, so that hypothesis identity remains consistent end-to-end.

#### Acceptance Criteria

1. THE Intelligence_Map endpoint `POST /api/intelligence/generate` SHALL accept an optional `hypothesis_manifest` parameter in the request body.
2. WHEN `hypothesis_manifest` is provided to `POST /api/intelligence/generate`, THE Intelligence_Map SHALL populate `explicit_hypothesis` nodes with a one-to-one mapping to the hypotheses in the manifest preserving hypothesis IDs.
3. WHEN `hypothesis_manifest` is provided to `POST /api/intelligence/generate`, THE Intelligence_Map SHALL NOT generate hypotheses from the intent string.
4. THE Intelligence_Map SHALL reserve `suggested_hypothesis` nodes for emergent hypotheses surfaced mid-study from signal anomalies.
5. WHEN `hypothesis_manifest` is provided to `POST /api/intelligence/generate`, THE Intelligence_Map convergence engine SHALL evaluate manifest hypotheses against signal evidence rather than free-generate them.
6. WHERE `hypothesis_manifest` is absent from the request, THE Intelligence_Map SHALL retain its current behavior.

### Requirement 17: Latest-Run Endpoint Update

**User Story:** As a developer using the dev bypass, I want the latest-run endpoint to include the cached manifest, so that downstream phases can be tested without re-running the engine.

#### Acceptance Criteria

1. THE Outtlyr_Platform SHALL include a `hypothesis_manifest` field in the response body of `GET /api/latest-run`.
2. THE Outtlyr_Platform SHALL load the value of `hypothesis_manifest` from `backend/latest_run_data/latest_hypothesis_manifest.json` when present.
3. WHERE `latest_hypothesis_manifest.json` is absent on disk, THE Outtlyr_Platform SHALL set `hypothesis_manifest` to null in the response.

### Requirement 18: Dev Bypass Updates

**User Story:** As a developer, I want the dev bypass buttons to handle the new phase, so that I can skip to any phase without manually triggering the engine.

#### Acceptance Criteria

1. WHEN the user clicks the "Skip to Synthesis" dev bypass button, THE Outtlyr_Platform SHALL load the cached `latest_hypothesis_manifest.json` and SHALL transition the Phase_State_Machine directly to `synthesis`.
2. THE Outtlyr_Platform SHALL provide a "Skip to Hypothesis Review" dev bypass button that loads the cached `latest_intent.txt` and `latest_hypothesis_manifest.json` and SHALL transition the Phase_State_Machine to `hypothesis_generation`.
3. WHEN the user clicks the "Skip to Discovery" dev bypass button, THE Outtlyr_Platform SHALL load the cached `latest_hypothesis_manifest.json` into state and SHALL transition the Phase_State_Machine to `discovery`.

### Requirement 19: Synthesis Dashboard Manifest Propagation

**User Story:** As a research analyst, I want the Synthesis_Dashboard to forward the manifest to all four synthesis API calls, so that all four artifacts are generated from the same hypothesis source.

#### Acceptance Criteria

1. THE Synthesis_Dashboard SHALL accept a `hypothesisManifest` prop in addition to the existing `interactionPayload` prop.
2. WHEN the Synthesis_Dashboard initiates synthesis, THE Synthesis_Dashboard SHALL pass the `hypothesisManifest` value into each of the four synthesis API calls.

### Requirement 20: Session Storage Caching

**User Story:** As a developer, I want the manifest cached in browser session storage keyed by intent hash, so that hot-reloads during development do not trigger redundant LLM spend.

#### Acceptance Criteria

1. WHEN the Hypothesis_Review_Screen receives a Hypothesis_Manifest from the API, THE Outtlyr_Platform SHALL write the manifest into browser `sessionStorage` keyed by a hash of the intent string.
2. WHEN the Hypothesis_Review_Screen mounts and a manifest is present in `sessionStorage` keyed by the current intent hash, THE Outtlyr_Platform SHALL load the cached manifest instead of invoking `POST /api/generate-hypotheses`.
